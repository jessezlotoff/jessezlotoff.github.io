Census Participation

by: Jesse Zlotoff

Research Goal

Census results are used to determine how federal money is allocated. Although
responding to the Census is required by law, not everyone participates. The
goal is to determine which demographic and geographic factors influence
responsiveness.

I set out to build a model that predicts response rates at the county level for
the 2010 decennial Census. That model could then be applied to recent demographic
data in order to predict responses to the upcoming 2020 Census (ignoring the
change to conducting some surveys online and complications due to the novel
coronavirus).

Data

The Census Bureau publishes participation rates for the 2010 Census. The rate
is defined as the percentage of questionnaires mailed back by households that
received them (overview here).

The 2009 American Community Survey (ACS) 5-year estimates provide demographic,
geographic and other information at the county level (list of variables here).
I use a subset of these variables, accounting for differences in county
population.

Additionally, I use 2006 county level urban-rural classifications from the
National Center for Health Statistics (NCHS) (overview here).

Results

The model predicts participation at the county level within a few points.

Holdout RMSE: 8.2%
Holdout R2: 0.70
Baseline RMSE: 11.2%

County size, income and other socio-economic features were the most predictive.

Modeling Process

After gathering the data, I did extensive cleaning of the ACS data. There are
over 20,000 raw variables split across multiple files. Key variables were
combined and then divided by the total population to make them comparable.

The modeling process following these general steps:

  1. Prepare the data
    1. Create a representative holdout set
    2. Do a train/test split on the remaining data
    3. Build interactions on all input variables
  2. Train, test and cross-validate models
    1. Build a baseline model from the mean of the target variable
    2. Try different model types: Lasso, Random Forest, Gradient Boosted Tree
      1. Train model
      2. Predict on test set
      3. Cross-validate
      4. Store the results
  3. Compare models and tune the features
    1. Compare model performance
    2. Select best model, using cross-validated root mean squared error (RMSE)
    3. Rank features by importance
    4. Tune number of features in best model
  4. Freeze model and run on holdout set
    1. Select best number of features
    2. Save model objects
    3. Predict and evalute on holdout set

Feature Tuning

I tested the best model with different sized subsets of features to improve
efficiency and help avoid overfitting. 50 features proved to be the best option.

Potential Outliers

In the comparison of modeled and true values above, it's clear that the model
has more trouble predicting very low target values. There are fewer of these
low values in the data as shown by the relatively small size of the dots.
Additional analysis shows values below 0.4 to be outliers in the train and test
sets, accounting for about 1% of values.

As a result, none of the models tested were able to predict these values well.
However, the models perform well overall.
